PHASE 2 — SERVER LOGIC
------------------------------------------------------------
A) When invoice is FINALIZED:
1) Generate invoice_number sequentially:
   - Format: INV-000001
   - Must be atomic; use a simple "invoice_sequences" table or DB sequence.
2) Set due_date:
   - due_date = period_end + grace_days (from clinic_billing_settings)
   - If grace_days missing => 0
3) Set:
   - payment_status='unpaid'
   - amount_paid_cents=0
   - balance_due_cents=total_cents

B) Payment status update function (single source of truth):
updateInvoicePaymentStatus(invoiceId):
- sumPayments = SUM(invoice_payments.amount_cents)
- amount_paid_cents = sumPayments
- balance_due_cents = max(total_cents - sumPayments, 0)
- if balance_due_cents == 0 => payment_status='paid'
- else if amount_paid_cents > 0 => payment_status='partial'
- else payment_status='unpaid'
- if now > due_date and payment_status != 'paid' => payment_status='overdue'
- set last_payment_at = max(paid_at)

This must run:
- after any payment insert
- when reading invoices list (apply overdue check lazily) OR via lightweight cron endpoint

C) Stripe Checkout creation:
- Create Stripe Checkout Session for the invoice balance due
- Line items: "Invoice INV-000123" (amount = balance_due)
- Metadata include: invoice_id, clinic_id, company_id, invoice_number
- Success URL: `${APP_PUBLIC_URL}/billing?paid=1&invoice=${invoiceId}`
- Cancel URL: `${APP_PUBLIC_URL}/billing?canceled=1&invoice=${invoiceId}`
- Store: stripe_checkout_session_id + stripe_checkout_url on invoice row

IMPORTANT:
- Use existing STRIPE_SECRET_KEY env var
- Ensure APP_PUBLIC_URL (or PUBLIC_BASE_URL) points to admin/app host.
- Allow both app/admin/driver origins in CORS already handled.

D) Stripe webhook (required):
- Endpoint: POST /api/stripe/webhook (rawBody already available)
- Validate Stripe signature header: Stripe-Signature using STRIPE_WEBHOOK_SECRET
- Handle events:
  - checkout.session.completed:
    - obtain session, payment_intent
    - determine paid amount (session.amount_total) or expand payment_intent
    - insert invoice_payments row:
      method='stripe' (or 'ach' if payment_method_types contains us_bank_account / bank_transfer)
      reference=session.id
      stripe_payment_intent_id
      paid_at=now
    - call updateInvoicePaymentStatus(invoiceId)
- Must be idempotent:
  - if a payment row already exists for stripe_payment_intent_id or session.id, do nothing.

PHASE 3 — API ENDPOINTS (ROLE GATED)
------------------------------------------------------------
Roles allowed (default):
SUPER_ADMIN, ADMIN, DISPATCH, COMPANY_ADMIN
Clinic_user:
- can view invoices for their clinic
- can download invoice PDF
- can create Stripe checkout link for their own invoice ONLY (optional; if unsure, restrict to dispatch/admin only)

Endpoints:

1) POST /api/invoices/:id/create-checkout
- Auth required
- Ensure invoice finalized and balance_due > 0
- Returns { url }
- Writes stripe_checkout_url to invoice

2) POST /api/invoices/:id/register-manual-payment
Body: { amountCents, method: 'manual', reference? }
- Inserts invoice_payments row
- Updates status

3) GET /api/invoices/:id
- Include payments summary: amount_paid, balance_due, payment_status, due_date

4) GET /api/invoices/:id/payments
- List payments rows

5) GET /api/invoices/:id/pdf
- Formal invoice PDF (NOT trip details PDF):
  Include:
  - UCM header + clinic name
  - invoice_number, issued date, due date
  - billing period
  - items table (trip lines + adjustments)
  - subtotal/fees/tax/total
  - amount paid + balance due
  - payment status
  - footer with support email/phone
- Headers:
  - Content-Type: application/pdf
  - Content-Disposition: attachment; filename="invoice_<INV-000123>.pdf"
  - Cache-Control: no-store

6) GET /api/reports/aging?asOf=...
- Returns buckets per clinic:
  - current (not due)
  - 1-30 overdue
  - 31-60
  - 61-90
  - 90+
- Use due_date and balance_due
- Role gate: SUPER_ADMIN/ADMIN/DISPATCH/COMPANY_ADMIN

7) POST /api/ops/recompute-overdue (optional utility)
- Walk invoices unpaid/partial and mark overdue if now>due_date
- Only SUPER_ADMIN/DISPATCH
(If you already have scheduler infrastructure, hook it; if not, keep as manual ops endpoint.)
