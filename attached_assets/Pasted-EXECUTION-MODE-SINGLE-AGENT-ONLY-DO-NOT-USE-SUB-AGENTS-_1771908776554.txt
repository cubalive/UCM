OBJETIVO (UNA SOLA TAREA): Normalizar el manejo de hora/fechas en toda la plataforma para que TODO coincida con la hora real de la ciudad operativa.
- Guardar timestamps en UTC en DB.
- Cada CITY debe tener timezone IANA.
- Todos los inputs de fecha/hora se interpretan como hora local de la ciudad y se convierten a UTC antes de guardar.
- Todos los outputs se muestran en hora local de la ciudad (driver / clinic / dispatch / admin).
- El trip guarda un snapshot del timezone usado (trip_timezone) para consistencia histórica.
- Resolver problemas actuales: hora de creación de viaje mal, horarios de pickups/dropoffs mal, dashboards con hora distinta, etc.

RESTRICCIONES:
- No romper datos existentes.
- No hacks con offsets manuales.
- Soportar DST automáticamente.
- No usar “new Date()” para mostrar horas sin pasar por conversión explícita.
- Mantener seguridad y performance.

   - trips: created_at, pickup_at, dropoff_at, scheduled_at, etc.
   - events/logs: created_at, occurred_at
3) Confirmar tipos en Postgres:
   - Deben ser timestamptz (timestamp with time zone) o al menos tratarse como UTC.
4) Identificar fuente de “ciudad” para cada pantalla:

PASO 1 — MODELO DE DATOS (DB)
A) Cities:
- Asegurar tabla cities tenga:
  - id
  - name
  - state
  - timezone (TEXT) // IANA, ej. "America/Los_Angeles"
- Backfill para ciudades existentes (si hay Las Vegas -> America/Los_Angeles, Dallas -> America/Chicago, etc.)

B) Trips:
- Agregar columnas:
  - trip_timezone TEXT NOT NULL DEFAULT (resolver por city.timezone)
  - pickup_at_utc timestamptz (o usar el mismo pickup_at pero garantizando UTC)
  - dropoff_at_utc timestamptz (si aplica)
  - scheduled_at_utc timestamptz (si aplica)
NOTA: Si ya existen pickup_at etc en timestamptz, NO dupliques; en ese caso:
- Mantén pickup_at como UTC y renómbralo lógicamente (no físicamente) y solo agrega trip_timezone.

C) Migración segura:
- Crear migration SQL con:
  - ALTER TABLE cities ADD COLUMN IF NOT EXISTS timezone text;
  - ALTER TABLE trips ADD COLUMN IF NOT EXISTS trip_timezone text;
  - Backfill: trips.trip_timezone = cities.timezone vía join (según trip.city_id).
- Validar que no hay nulls finales.

PASO 2 — “TIME SERVICE” ÚNICO (SERVER + CLIENT)
Elegir una librería consistente:
- Recomendado: luxon (IANA + DST fácil) o date-fns-tz.
Implementar helpers compartidos (ideal en /shared):
1) toUtcFromCityLocal(localISO, cityTz) -> returns ISO UTC
   - localISO es "YYYY-MM-DDTHH:mm" SIN timezone (lo que sale del datepicker)
2) toCityLocalFromUtc(utcISO, cityTz) -> returns display string + local ISO
3) formatCity(utcISO, cityTz, format) -> string
4) nowInCity(cityTz) -> display + epoch

Prohibir en UI:
- toLocaleString sin timezone explícito
- new Date(utc).toString() sin conversión

PASO 3 — APIs: LADO SERVIDOR (FUENTE DE VERDAD)
A) Resolver timezone por contexto:
- Crear función server/utils/timezone.ts:
  - getCityTimezone(cityId): fetch cities.timezone
  - getActorTimezone(actor): según rol:
    - driver -> driver.city_id timezone
    - clinic -> clinic.city_id timezone
    - dispatch/admin -> si seleccionó city filter, usar esa; si no, fallback company default.

B) En endpoints de creación/edición de trip:
- Si el cliente manda pickup_local + city_id:
  - tz = city.timezone
  - pickup_utc = toUtcFromCityLocal(pickup_local, tz)
  - guardar pickup_at = pickup_utc
  - guardar trip_timezone = tz (snapshot)
- created_at debe ser DEFAULT now() en DB (UTC).

C) En endpoints de lectura:
- Enviar SIEMPRE:
  - pickup_at (UTC ISO)
  - trip_timezone

Recomendación: enviar UTC + trip_timezone y UI formatea, pero con helpers centralizados.
