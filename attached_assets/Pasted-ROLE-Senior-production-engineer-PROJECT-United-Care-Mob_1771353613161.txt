PHASE 1 — ARCHIVE / UNARCHIVE (SOFT DELETE)
========================================================
Definition:
- Archive: active=false AND deletedAt=now()
- Unarchive: active=true AND deletedAt=NULL

Entities and who can do what:
1) clinics:
- SUPER_ADMIN: archive/unarchive any
- DISPATCH/COMPANY_ADMIN: archive/unarchive within tenant & allowed city
- CLINIC_USER/DRIVER: no

2) drivers:
- SUPER_ADMIN: any
- DISPATCH/COMPANY_ADMIN: within tenant & allowed city
- CLINIC_USER/DRIVER: no

3) patients:
- SUPER_ADMIN: any
- DISPATCH/COMPANY_ADMIN: within tenant & allowed city
- CLINIC_USER: archive/unarchive ONLY patients in their clinic (patient.clinicId must equal actor.clinicId)
- DRIVER: no

Active-trip guard (must block archive):
- For clinic: if any trips with clinicId and status in ('SCHEDULED','ASSIGNED','IN_PROGRESS') and deletedAt IS NULL -> 409
- For driver: same by driverId -> 409
- For patient: same by patientId -> 409

Audit:
- Create auditLog entry for every archive/unarchive:
  eventType: archive_* / unarchive_*
  entityType: clinic/driver/patient
  entityId
  actorUserId, actorRole, companyId, cityId
  metadata JSON before/after { active, deletedAt }

PHASE 2 — HARD DELETE (SUPER_ADMIN ONLY, CONFIRM WORD)
========================================================

A) Create a single generic endpoint (preferred):
POST /api/admin/hard-delete
Body:
{
  "entity": "company" | "city" | "clinic" | "driver" | "patient" | "trip" | "invoice" | "billingCycleInvoice" | "vehicle" | ...
  "id": number,
  "confirmWord": "delete"
}

Rules:
1) requireAuth + requireRole("SUPER_ADMIN") ONLY
2) If confirmWord !== "delete": return 400 { error: "Confirmation required. Type 'delete'." }
3) Before delete, run safety checks and show clear errors:
   - If entity has active trips -> 409 (unless deleting trip itself)
   - If entity has dependencies, delete in safe order or refuse with instructions.

B) Delete order (MUST be safe, no orphan breaks):
- For patient delete:
  delete related: tripSmsLog, tripShareTokens, tripEvents, tripSignatures, tripBilling, invoice lines/items tied to patient trips, then trips, then patient
- For driver delete:
  close assignment history, delete offers/alerts/tokens, unassign from trips if needed (or block if active trips), then delete driver
- For clinic delete:
  block if active trips; otherwise delete invoices/billing profiles/items, tariffs, clinic users (or detach), patients (only if admin confirms per-entity) OR implement cascade delete path with clear logging.
- For company delete:
  highest risk. MUST implement preflight summary first OR block unless "force=true" + confirmWord. (Implement preflight v1 in this task.)

C) Add PREVIEW endpoint (must exist before actual destructive deletes for company/city):
GET /api/admin/hard-delete/preview?entity=company&id=...
Returns counts of dependent rows (clinics, drivers, patients, trips, invoices, etc).
Only SUPER_ADMIN.

D) Audit hard deletes:
- eventType: hard_delete
- metadata includes entity, id, dependentCounts (from preview), confirmWordUsed, deletedBy

E) UI:
- In SUPER_ADMIN settings add "Danger Zone" modal:
  - Input box requiring typing: delete
  - Show preview counts
  - Only enable confirm if exact match.
