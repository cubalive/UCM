PHASE 2 — RBAC + TENANT SCOPING (this is critical)
Problem: Many storage methods query only by cityId/clinicId etc and do NOT consistently filter by companyId. This allows cross-company data leakage if multiple companies exist.

Implement and enforce:
A) requireRole() stays, but remove/avoid normalizeRole() hacks that map CLINIC_USER to VIEWER. Roles must be explicit and consistent.
B) Add middleware:
- requireAuth (already authMiddleware)
- requireCompanyScope: if user is not SUPER_ADMIN, companyId must exist for COMPANY_ADMIN and must be applied in queries.
- requireCityAccess: for non SUPER_ADMIN, enforce user_city_access for any city-scoped read/write.
C) Update storage layer to accept (companyId, cityIds) in relevant methods OR create scoped wrappers:
Example: getClinics(cityId, companyId) -> WHERE clinics.cityId=... AND (clinics.companyId=companyId OR clinics.companyId IS NULL)
Do this for: clinics, patients, drivers, vehicles, trips, invoices, tariffs, billing.

Acceptance:
- A COMPANY_ADMIN cannot fetch clinics/patients/trips from another company.
- A DISPATCH user can only access allowed cities (user_city_access).
- SUPER_ADMIN bypass works.

PHASE 3 — SOFT DELETE CONSISTENCY (stop hard deletes in production flows)
Problem: schema supports deletedAt/deletedBy/deleteReason but delete*() uses db.delete() in storage for many entities.
Implement consistent soft-delete:
- storage.softDeleteClinic/Driver/Patient/User/Trip/Vehicle: set deletedAt=now, deletedBy=userId, deleteReason=reason, active=false where relevant
- Keep existing hard delete only for SUPER_ADMIN maintenance (optional) but default API uses soft delete.
- Ensure all list queries filter deletedAt IS NULL.

Acceptance:
- Deleting a clinic marks it deleted (not removed). Archived views show it.
- Existing foreign keys do not break.
- Trips for deleted clinic remain but are still readable by authorized roles.
