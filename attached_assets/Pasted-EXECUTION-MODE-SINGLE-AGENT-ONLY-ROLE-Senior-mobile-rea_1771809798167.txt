GOAL
1) Driver accepts future trip -> it is RESERVED (upcoming) and must NOT make driver busy.
2) Trip becomes "active" and driver becomes BUSY only at dispatch_at.
3) Notifications must be based on ETA/dispatch windows (notify_at + dispatch_at).
4) If tracking is lost (no location ping) for >60s, system alerts driver and flags to admin/dispatcher/clinic.
5) Make it Capacitor-ready: add hooks and config for background location (native) while keeping PWA fallback.

STEP 1 — Driver UI behavior
Implement in driver client:
- "Upcoming Reservations" list (accepted future trips)
- Hide reserved trip from active screen until now >= dispatch_at
- When now >= notify_at: show banner "Trip soon" + haptics
- When now >= dispatch_at: move trip to Active Trip screen automatically

STEP 2 — Backend dispatch trigger + idempotent notifications
In scheduler (every 30-60s):
For each trip assigned and not terminal:
- if now >= notify_at and dispatch_stage='NONE':
  emit WS event to driver + optional push/SMS: "Prep now" (do not change trip status)
  set dispatch_stage='NOTIFIED'
- if now >= dispatch_at and dispatch_stage IN ('NONE','NOTIFIED'):
  set driver.availability_status='BUSY'
  transition trip status -> EN_ROUTE_TO_PICKUP (use existing transition function)
  emit WS event "DISPATCH_NOW"
  set dispatch_stage='DISPATCHED'

All must be idempotent (retries safe).

STEP 3 — Tracking health (loss detection)
Driver app sends location ping every 10-15 seconds when in foreground.
Backend stores:
- drivers.last_location_at, lat, lng
Every 30s scheduler:
- if now - last_location_at > 60s:
   set drivers.tracking_status='STALE'
   emit WS alert to driver: "Open app to keep tracking"
   emit WS alert to dispatcher/admin/clinic for any active trips: "Tracking paused"
- if ping resumes:
   set tracking_status='OK' and emit "Tracking restored"
