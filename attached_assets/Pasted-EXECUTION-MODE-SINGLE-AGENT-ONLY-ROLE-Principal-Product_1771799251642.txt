PHASE 2 — LEADER ELECTION + SCHEDULER LOCKS
========================
Even with a worker, we must be safe if multiple workers run.

Implement a Redis-based leader lock:
- key: ucm:worker:leader
- value: instanceId
- SET NX EX 30
- renew every 10s while alive
- if lock lost, STOP all schedulers immediately

Per-scheduler lock as defense-in-depth:
- key: ucm:scheduler:<name>
- SET NX EX <interval-5>
- prevents double-execution if leader glitch happens

MUST:
- If Redis not configured -> schedulers run in SINGLE instance only with loud warning
- In prod, require Redis (fail fast if missing) unless ALLOW_NO_REDIS=true

PHASE 3 — REAL JOB QUEUE (DURABLE) ON REDIS
========================
Your current jobQueue.ts exists; harden it to enterprise:
- Queue lists per type: ucm:queue:<type>
- In-flight set: ucm:inflight:<type>
- DLQ: ucm:dlq:<type>
- Job payload includes:
  jobId (uuid)
  type
  tenantId/cityId scope
  dedupeKey
  createdAt
  runAt
  attempts
  maxAttempts
  backoffMs
  traceId

Processing:
- Pop -> move to inflight (atomic)
- Ack on success (remove inflight)
- On fail: attempts++ and requeue with exponential backoff
- After maxAttempts -> DLQ
- Add admin endpoint to inspect queues (SUPER_ADMIN only)

Idempotency:
- Use Redis SETNX ucm:dedupe:<dedupeKey> EX 86400 before enqueue/execute
- If dedupe exists -> skip duplicate

Convert every scheduler loop into enqueue-only:
- Scheduler does NOT perform heavy work.
- Scheduler enqueues jobs (e.g., ETA_RECALC, AUTO_ASSIGN, NO_SHOW_SCAN, ROUTE_REFRESH, PAYROLL_RUN, DUNNING_RUN, DIALYSIS_GEN, SMS_REMINDER)
- Worker processors execute jobs.
