PHASE C — ENDPOINTS (EXTEND IMPORT V1)
==================================================

Add entity "trips" and "billing" support using existing endpoints:
- upload?entity=trips (CSV/XLSX/JSON)
- upload?entity=billing (optional; can be included in same file or separate)

Add/extend:
POST /api/admin/imports/:id/validate
- Parse trips + billing files
- Map to canonical
- Validate:
  - clinic resolvable (via external_id_map OR name match)
  - patient resolvable (via external_id_map OR strong natural match)
  - scheduled_pickup_at present
  - addresses present
- Dedupe in-file by external_trip_id OR (patient+pickup_at+pickup_address)
- Build preview and error report

POST /api/admin/imports/:id/run
- Execute import in order (already v1 did clinics/patients/drivers/vehicles):
  e) trips
  f) billing (trip_billing/invoices linkage)
- Trip upsert / idempotency:
  1) If external_trip_id present and exists in import_trip_idempotency -> update that trip safely
  2) Else attempt match:
     - same patient + scheduled_pickup_at (+/- 10 min) + pickup_address normalized
  3) If match -> update and create idempotency row if external_trip_id exists
  4) If no match -> insert trip and create idempotency row

- Billing application rules (NO external APIs):
  - If BillingCanonical.total_cents provided -> store in trip_billing.totalCents and/or trips.price_cents (depending existing design; do NOT break)
  - Else if distance_miles present and clinic_tariffs exists:
      compute price estimate using active tariff (base + per-mile + wheelchair extra + wait)
      store in trip_billing and trips.price_cents (if current flow expects it)
  - Else leave pricing_missing and pricing_required true (safe), log warning

- Invoice handling (minimal, safe v2):
  - Do NOT auto-finalize invoices.
  - If your system supports invoice grouping tables, create draft invoices only when:
      a) clinic_id present AND
      b) trip status is COMPLETED or marked billable AND
      c) enough pricing data exists
  - Otherwise, skip invoice creation and log “invoices_missing_count” relevance for intelligence.

- Record import_job_events per trip:
  - created/updated/skipped + reasons

POST /api/admin/imports/:id/rollback
- Roll back only trips created by this job:
  - Track created trip IDs in import_job_events payload
  - Delete ONLY those trips if they have no downstream locks (e.g., signatures or payments). If locked, mark cancelled and log.
- Roll back created trip_billing rows for those trips if they were created by this job.
