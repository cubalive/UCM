B) Invoices UI:

REQUIREMENTS

1) UNIVERSAL CLICK-THROUGH STANDARD
Implement a shared component or utility:
- <ClickRow to="/trips/:id"> or similar
- “TripRef” component: renders trip display (Trip #, patient, pickup time) and links to /trips/:tripId
- “InvoiceRef” component links to /invoices/:invoiceId
Use it everywhere a trip/invoice appears (no duplicates, no one-off link hacks).

2) TRIP DETAILS MUST WORK FOR ALL STATUSES
Fix the root issue: completed trips (or billed trips) currently cannot be opened from invoices.
- Ensure GET /api/trips/:id returns trips regardless of status (active/completed/canceled/no-show), as long as the user has permission.
- Ensure list queries don’t exclude completed trips when loading invoice context.
- If there is “activeOnly” filtering, make it explicit and never used by default on detail views.

3) INVOICE → TRIP DRILLDOWN
On invoice detail:
- Each trip line item must link to Trip Detail.
- Trip Detail must show “Billed / Invoice #” metadata and link back to invoice.
- If tripId is missing and only an externalRef exists, implement a resolver:
  - /api/trips/resolve?externalId=... OR resolve by invoice_line_item.trip_id mapping.
  - Must return the internal UUID tripId reliably.

4) RBAC + TENANT SAFETY
- SUPER_ADMIN: can open any trip/invoice
- COMPANY_ADMIN: only trips/invoices for their company
- CLINIC_USER: only trips tied to their clinic + company rules
- DRIVER: only trips assigned to them (historical included)
No cross-tenant leakage.

5) UX SPEC

- Also keep explicit “View” icon/button for accessibility.

- On mobile: tap area must be large; no tiny links.
