1) DEPLOYMENT MODEL (CRITICAL)
========================================================
We will run TWO deployments/services:

A) API Service (stateless)
- Handles HTTP requests only
- Enqueues jobs to Redis
- NEVER generates PDFs/invoices in-request
- Can scale horizontally without shared memory

B) Worker Service (background)
- Runs continuously
- Pulls jobs from Redis
- Generates PDFs / invoices / emails / billing rollups
- Writes results to DB and/or object storage
- Updates job status

Implement both in same repo with two entrypoints:
- api server:  src/server.ts (or existing)
- worker runner: src/worker.ts

Add scripts:
- "start:api" -> start API server
- "start:worker" -> run worker processor loop

========================================================
2) HARD MULTI-TENANT ENFORCEMENT
========================================================
Implement middleware:
- Resolve user + role + company_id from JWT
- company_id REQUIRED for all non-super-admin routes
- Inject ctx.companyId
- Add DB query guard helper that REQUIRES company_id filters for any tenant table:
  trips, drivers, vehicles, clinics, invoices, schedule_change_requests, etc.

Fail closed:
- If company_id missing -> 401/403
- If query attempts cross-company -> 403 + log system_event

Redis keys MUST be tenant-prefixed:
company:{companyId}:...

3) JOB QUEUE (REDIS) + DLQ
========================================================
Implement Redis-backed queue with:
- enqueue(type, payload, opts)
- lease/lock with visibility timeout
- retries with exponential backoff
- max attempts
- dead-letter queue (DLQ) recording last error

Job types:
- pdf_trip_details
- invoice_generate
- billing_rollup
- email_send
- map_snapshot

DB table: jobs
- id (uuid)
- company_id
- type
- status: queued|working|succeeded|failed
- attempts
- priority
- payload jsonb
- result jsonb (url, metadata)
- last_error text
- created_at, updated_at

API endpoints:
- POST /api/jobs (internal helper if needed)
- GET  /api/jobs/:id  (status + result)
- GET  /api/trips/:id/pdf -> MUST enqueue pdf_trip_details and return 202 + jobId
- GET  /api/trips/:id/pdf/download -> streams completed PDF only if job succeeded AND user authorized

The worker must:
- pick job
- mark working
- process
- store result (preferred: store file + return URL; if no storage exists, store PDF bytes in DB as bytea temporarily, but cap size and plan migration to storage later)
- mark succeeded/failed
