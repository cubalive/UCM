DRIVER REQUEST ACCEPTANCE (CRITICAL):
Goal: When dispatch sends an assignment/request to a driver, the request remains visible/active for an acceptance window and does not disappear quickly.

Backend rules:
A) Define request TTL:
   - Default: 90 seconds (configurable; can be in city_settings or env).
B) If an assignment is offered:
   - Persist an "offer" record with:
     offer_id, trip_id, driver_id, offered_at, expires_at, status('pending','accepted','expired','cancelled')
   - If you already have a mechanism for dispatch->driver requests, extend it with expires_at + status and enforce TTL.
C) Endpoint behavior:
   - Driver polling endpoint must return pending offers that have not expired.
   - Do NOT remove pending offers from UI until:
     1) driver accepts, OR
     2) expires_at reached (then mark expired server-side), OR
     3) dispatch cancels/reassigns (mark cancelled).
D) Acceptance validation:
   - POST /api/driver/offers/:offer_id/accept must:
     - verify offer belongs to this driver
     - verify status == 'pending'
     - verify now() <= expires_at
     - then mark accepted and assign trip to driver (atomic transaction).
E) UI countdown:
   - Show countdown timer on the request card:
     "Expires in 01:29"
   - When expired, show "Request expired" and remove after 3 seconds (or move to a small “Expired” section).
