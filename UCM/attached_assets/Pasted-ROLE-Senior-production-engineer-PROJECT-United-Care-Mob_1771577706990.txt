========================================
C) BILLING AUTOMATION + STRIPE PAYMENTS (WEEKLY)
========================================
You must design this cleanly (no hacks), but implement in incremental safe steps.

BUSINESS RULES
- Clinics/Hospitals pay Companies weekly (invoice-based).
- Private patients can pay per trip or weekly invoice (configurable per company).
- Drivers are paid hourly (based on approved timecards) by the Company weekly.
- Platform (UCM) may take a configurable fee % (already exists “Platform Fees” screen) from Stripe payments.

STRIPE ARCHITECTURE (recommended)
Use Stripe Connect:
- Each Company = Stripe Connect Account (Express or Standard)
- Each Driver who receives payouts = Stripe Connect Account (Express)
- Clinics/Private patients pay via Checkout/Invoice/PaymentIntent.
- Money flow:
  - Clinic → (Stripe) → Company (destination charge or separate charge+transfer)
  - Company → Driver payout (transfer to driver connected account, or Stripe payouts via driver account)
- Platform fee:
  - application_fee_amount (or application_fee_percent if using Invoices with Connect)
  - Track fees in DB for audit.

DATA MODEL (additive, no breaking changes)
Create new tables if they don’t exist (Drizzle migrations):
1) billing_profiles
- id, company_id, billing_cycle ('weekly'), timezone, default_terms_days, enabled
2) clinic_billing_configs
- id, clinic_id, company_id, weekly_rate_rules (json), invoice_day_of_week, enabled
3) invoices (if already exists, extend carefully)
- add: billing_period_start, billing_period_end, stripe_invoice_id, stripe_payment_intent_id, payment_status, totals, currency
4) invoice_line_items
- id, invoice_id, trip_id nullable, description, qty, unit_amount, amount, metadata json
5) driver_pay_profiles
- id, driver_id, company_id, hourly_rate, overtime_rules json, enabled
6) payroll_runs
- id, company_id, period_start, period_end, status (draft/approved/paid), totals
7) payroll_items
- id, payroll_run_id, driver_id, hours, rate, amount, stripe_transfer_id, payout_status
8) stripe_accounts
- id, owner_type ('company'|'driver'), owner_id, stripe_account_id, onboarding_status, charges_enabled, payouts_enabled
9) jobs
- id, type, scope, status, run_at, started_at, finished_at, error, attempts, idempotency_key

WEEKLY AUTOMATION
1) Weekly Clinic Invoices
- For each company that has billing enabled:
  - For each clinic with weekly billing enabled:
    - Determine billing period (Mon–Sun or configurable)
    - Collect billable trips in that range (completed, not already invoiced)
    - Generate invoice draft in DB + line items referencing trips
    - Provide Preview Trips list (already exists UI) but also allow “Auto-generate”
2) Payment Collection
- Provide “Pay Now” for clinics:
  - If using Stripe Invoices: create Stripe invoice with line items, finalize, send invoice OR create hosted invoice URL.
  - If using Checkout Session: create session for invoice amount and redirect.
- Store stripe ids + webhook updates (paid/failed).
3) Private Patient Payments
- For trips flagged as private-pay:
  - On trip completion, generate a payable item (single trip invoice or add to weekly private invoice).
  - Offer Pay link to patient (SMS/email) later; store payment status.
4) Driver Payroll (Hourly)
- Timecards are already present: rely on “Approved” entries.
- Weekly payroll run:
  - Period start/end
  - Aggregate approved hours by driver
  - Calculate amounts using driver hourly_rate + rules
  - Create payroll_run + payroll_items
- Payout via Stripe:
  - Company must have Connect enabled and sufficient balance OR use separate charge model.
  - For initial version: implement “Mark as Paid” + “Send payout” (Stripe transfer) when driver has connected account.
  - Track failures and show in Ops Health.

WEBHOOKS + SECURITY
- Implement Stripe webhooks endpoint:
  - invoice.paid, invoice.payment_failed (if using invoices)
  - checkout.session.completed (if using checkout)
  - account.updated (connect onboarding)
- Verify Stripe signature.
- RBAC:
  - SUPER_ADMIN can see all.
  - Company Admin sees only their company billing + payroll.
  - Clinic sees only their invoices and pay links (no company-wide data).
  - Driver sees only their timecards + payroll history.

ACCEPTANCE TESTS (C)
- Company admin can enable billing profile and connect Stripe account (onboarding link).
- Clinic billing: generate weekly invoice draft → preview trips list → finalize → Pay Now opens Stripe hosted page.
- After payment, invoice status updates to PAID via webhook.
- Payroll: generate payroll run from approved timecards → shows per-driver hours & amount → payout attempts create Stripe transfers for drivers who completed onboarding.

D) UPSTASH (REDIS) FOR JOBS + IDEMPOTENCY
========================================
Use Upstash Redis for:
- distributed locks (avoid double-running weekly invoice generation)
- idempotency keys (stripe calls, invoice generation)
- job state caching (optional)

IMPLEMENTATION
1) Add env vars (do not break existing):
- UPSTASH_REDIS_REST_URL
- UPSTASH_REDIS_REST_TOKEN
(If these already exist, use them.)

2) Job runner
- Implement a small job system:
  - enqueueJob(type, payload, runAt, idempotencyKey)
  - runDueJobs() runs periodically (every 1–5 minutes) in production
- Use Redis lock: lock:job_runner with TTL 60s so only one instance runs.
- Each job row in DB stores attempts and last error.
- If Upstash QStash is available in this project, you may add it later; for now Redis lock + DB jobs is enough.

3) Ops Health integration
- Show Billing Jobs + Payroll Jobs status and failure rate
- Clicking takes user to job list filtered by failed jobs.

ACCEPTANCE TESTS (D)
- Trigger “Run weekly billing now” → creates a job, locks prevent duplicates.
- Repeated clicks do not create duplicate invoices (idempotent).
- Failures appear in Ops Health and are clickable to the job detail.
