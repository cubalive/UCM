PHASE 1 — BACKEND: INVOICE ENDPOINTS
---------------------------------------
Implement these endpoints (or equivalent routes if your server structure differs):

A) GET /api/trips/:id/invoice
- Returns invoice if trips.invoice_id exists OR invoice found by invoices.trip_id = trip.id
- If found by trip_id but trips.invoice_id is null, set trips.invoice_id to that invoice.id (safe backfill)

B) POST /api/trips/:id/invoice (Create/Attach)
- Allowed roles: SUPER_ADMIN, DISPATCH
- Preconditions:
  - Trip must be COMPLETED (only completed can generate billable invoice)
  - Trip must belong to same company_id as user (if enforced)
- Behavior:
  - If invoice already exists, return it (idempotent)
  - Else create invoices row using:
    - clinic_id from trip.clinic_id (required)
    - trip_id = trip.id
    - patient_name from patients first+last (join via trip.patient_id)
    - service_date from trip.scheduled_date (or completed_at date if available; pick one and document)
    - amount from request body (required > 0)
    - status default 'pending'
    - notes optional
  - Set trips.invoice_id = created invoice id
  - Write audit_log entry: action='invoice_created', entity='trip', entity_id=trip.id, details includes invoice id and amount

C) PATCH /api/invoices/:id
- Allowed roles: SUPER_ADMIN, DISPATCH
- Update: amount, status (pending/approved/paid), notes
- Write audit_log entry: action='invoice_updated', entity='invoice', entity_id=invoice.id

D) POST /api/invoices/:id/pdf
- Allowed roles: SUPER_ADMIN, DISPATCH, CLINIC_USER (clinic can download their own invoice PDF)
- If pdf_url already exists and file exists, return url.
- Otherwise generate the PDF using the existing PDFKit implementation already added in billing.
- Store resulting pdf_url back to invoices.pdf_url.
- Return download url.

E) Security for clinic:
- Clinic user can only access invoices where invoice.clinic_id = users.clinic_id (or from auth context).
- Never allow clinic to update invoice fields (read/download only).
