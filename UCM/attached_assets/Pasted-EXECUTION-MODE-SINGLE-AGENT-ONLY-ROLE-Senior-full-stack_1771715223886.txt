STEP 1 — DATABASE (idempotent Supabase SQL)
--------------------------------------------
1) Create table driver_pay_rules (1 row per company, ON/OFF switches + params):
- id uuid PK default gen_random_uuid()
- company_id uuid not null unique
- daily_min_enabled boolean not null default false
- daily_min_cents integer null
- daily_min_applies_days text[] null  -- e.g. {'MON','TUE','WED','THU','FRI','SAT'}
- on_time_bonus_enabled boolean not null default false
- on_time_bonus_mode text null check in ('PER_TRIP','WEEKLY')
- on_time_bonus_cents integer null
- on_time_threshold_minutes integer null default 5
- on_time_requires_confirmed_pickup boolean not null default true
- no_show_penalty_enabled boolean not null default false
- no_show_penalty_cents integer null
- no_show_penalty_reason_codes text[] null
- created_at timestamptz default now()
- updated_at timestamptz default now()

2) Add columns to trips (if missing) to support punctuality and penalties:
- pickup_scheduled_at timestamptz null (or reuse existing)
- pickup_actual_at timestamptz null (arrived_pickup_at or picked_up_at fallback)
- is_no_show boolean not null default false
- no_show_reason text null

3) Create table driver_earnings_adjustments (auditable):
- id uuid PK default gen_random_uuid()
- company_id uuid not null
- driver_id uuid not null
- related_trip_id uuid null
- period_date date null          -- for daily minimum adjustments
- week_start date null
- type text not null check in ('DAILY_MIN_TOPUP','ON_TIME_BONUS','NO_SHOW_PENALTY','MANUAL_ADJUSTMENT')
- amount_cents integer not null  -- can be negative
- metadata jsonb not null default '{}'
- created_at timestamptz default now()

Indexes:
- driver_earnings_adjustments(company_id, driver_id, created_at)
- driver_earnings_adjustments(company_id, driver_id, week_start)

All migrations must be idempotent.

STEP 2 — BACKEND SERVICES (server/services/payroll/*)
--------------------------------------------
Create services:

A) payPolicyService.ts
- existing computeTripDriverPayout(tripId) remains source of base payout (no modifiers).

B) payrollRulesService.ts
- getCompanyPayRules(companyId)
- upsertCompanyPayRules(companyId, payload)

C) modifiersEngine.ts
Function:
computeTripModifiers(tripId) -> adjustments[]
Applies:
1) On-time bonus (if enabled)
   - Determine on-time:
     pickup_actual_at - pickup_scheduled_at <= threshold
     Use arrived_pickup_at or picked_up_at as actual (prefer arrived_pickup_at)
   - If on_time_bonus_mode=PER_TRIP: create adjustment ON_TIME_BONUS +amount
   - If WEEKLY: defer to weekly job (see below) unless existing bonus system already does weekly bonuses.
2) No-show penalty (if trip is_no_show=true and enabled):
   create NO_SHOW_PENALTY as negative amount_cents

D) dailyMinimumEngine.ts
computeDailyMinimumTopups(companyId, date) -> adjustments[]
- For each driver in company with on_shift activity or completed trips that day:
  dailyEarnings = sum(base payouts + per-trip modifiers) for that date
  if daily_min_enabled and dailyEarnings < daily_min_cents and date is in applies_days:
     create DAILY_MIN_TOPUP for difference

E) weeklySummaryService.ts
getWeeklyEarnings(driverId, weekStart)
Return:
- baseEarningsCents (sum trips.driver_payout_cents)
- modifiersCents (sum adjustments for week)
- totalCents = base + modifiers
- breakdown arrays (trip payouts + adjustments)

IMPORTANT:
- Ensure idempotency: never duplicate adjustments.
  Use a deterministic key in metadata (e.g. {key:"ON_TIME_BONUS:tripId"}), check before insert.

STEP 3 — WHEN TO APPLY MODIFIERS
--------------------------------------------
A) On trip completion:
- compute base payout and store on trip (already)
- computeTripModifiers(tripId) and upsert adjustments (idempotent)
- emit realtime event to update driver earnings UI

B) Daily minimum:
- run a scheduled job (existing scheduler pattern) at e.g. 23:59 local time per company city/timezone
- OR compute on-demand when driver opens earnings and cache results
Prefer scheduled job if system already has cron/scheduler.

C) Weekly on-time bonus (if enabled and mode=WEEKLY):
- at week end (Sunday 23:59) compute if driver meets threshold criteria:
   e.g. % on-time >= configured percent (add optional field if needed)
If existing bonus module already does weekly driver bonus:
- integrate by reading its outputs instead of duplicating logic.
