PHASE 2 — EARNINGS GENERATION (NON-BREAKING)
========================================
Implement earnings generation using existing trip lifecycle:
- For PER_TRIP mode:
  When trip is finalized (finalized_at set) OR status COMPLETED:
  create ledger row TRIP with amount = per_trip_flat OR percent of trip price (if price exists), else fallback per_trip_flat.
- For HOURLY mode:
  Use trip duration if available:
  duration_minutes = (actual_dropoff_at - actual_pickup_at) OR existing duration field OR 0.
  hours = duration_minutes / 60.
  amount = round(hours * hourly_rate_cents).
  create ledger HOURLY row per trip (v1) so no new timesheet UI is required.

Eligible rules:
eligible_at = earned_at + holdback_days
If require_trip_finalized=true: only generate when finalized_at not null.
If require_clinic_paid=true: only eligible when invoice marked paid (if available).

Ensure idempotency: do not duplicate ledger rows for same (trip_id, earning_type) per company+driver.

========================================
PHASE 3 — PAYRUN CREATION + APPROVAL
========================================
Endpoints (role-gated):
- GET  /api/company/payroll/settings (COMPANY_ADMIN + SUPER_ADMIN)
- PUT  /api/company/payroll/settings (COMPANY_ADMIN + SUPER_ADMIN)

- POST /api/company/payroll/payruns/generate?periodStart=YYYY-MM-DD&periodEnd=YYYY-MM-DD
  Creates a DRAFT payrun for that period:
  - selects ledger rows status ELIGIBLE for that company within eligible_at <= periodEnd
  - groups by driver, sums amount
  - applies minimum_payout_cents filter (below => keep ELIGIBLE, not included)
  - creates payrun + items
  - marks included ledger rows IN_PAYRUN with payrun_id

- POST /api/company/payroll/payruns/{id}/approve
  Changes status to APPROVED (COMPANY_ADMIN only for their company)

========================================
PHASE 4 — STRIPE PAYOUT EXECUTION
========================================
Execution endpoint:
- POST /api/company/payroll/payruns/{id}/process
  Auth: COMPANY_ADMIN (own company) or SUPER_ADMIN
  Behavior:
  - validates payrun APPROVED
  - for each item:
     - verify driver_stripe_accounts ACTIVE + payouts_enabled
     - create Stripe Transfer to driver connected account
     - store stripe_transfer_id, paid_at
  - set ledger rows PAID
  - set payrun status PAID if all succeeded else FAILED
Idempotency:
  - use payrun.idempotency_key with Stripe requests
  - if already PAID, return success without re-paying

========================================
PHASE 5 — SCHEDULER (AUTO PAY ON DUE DATE)
========================================
Create a server cron job endpoint (protected by internal secret):
- POST /api/internal/payroll/run-due
Logic:
  - find payruns with scheduled_payday = today and status APPROVED
  - process them
Schedule:
  - Use existing Replit cron / scheduled task mechanism.
  - Ensure runs once daily in each company timezone by computing “today” per timezone.
