PHASE 2 — DRIVER: ACTIVE / ON-HOLD + LOCATION PERMISSIONS
========================================================
Goal: driver must clearly indicate availability, and location must work.

1) Driver availability:
- Driver has status: ACTIVE or ON_HOLD
- Driver app has a big toggle:
  - “Go Active” / “Go On Hold”
- Backend endpoints:
  - POST /api/driver/status { status: "ACTIVE"|"ON_HOLD" }
  - GET /api/driver/me (includes status)
- Dispatch can override:
  - POST /api/dispatch/drivers/:driverId/status

2) Location:
- On driver app start, request geolocation permission.
- If denied, show blocking banner: “Location required to run trips” with retry button.
- Start sending location updates when driver is ACTIVE:
  - every 10–15 seconds while app is open OR every 20–30s if battery-saving
- Backend:
  - POST /api/driver/location { lat, lng, heading?, speed?, accuracy?, ts }
  - Store latest location per driver (table or in drivers table)
- Security:
  - Driver can only update their own location.
  - Dispatch/admin can read location for drivers in active city.
  - Clinic can read ONLY the assigned driver’s location for their trips.

PHASE 3 — DISPATCH: ASSIGN BUTTON + DAILY ASSIGNMENTS FIX
========================================================
Problem: approved trip cannot be assigned + clinic-created trips not showing in daily assignment.

1) Add “Assign Driver” button on every trip card in UNASSIGNED or SCHEDULED state (dispatch/admin).
- Opens modal:
  - Select driver (ACTIVE only, same city)
  - Optional vehicle selection (same city, correct type)
  - Confirm

2) Backend:
- POST /api/dispatch/trips/:tripId/assign { driverId, vehicleId? }
Validations:
- trip city_id == activeCityId
- driver city_id == activeCityId
- driver status == ACTIVE
- trip not COMPLETED/CANCELLED/NO_SHOW
- if wheelchair trip => require wheelchair-capable vehicle OR allow WAV vehicle for sedan (but never sedan for wheelchair)
On success:
- set status = ASSIGNED
- notify driver (in-app + optional SMS later)

3) Daily Assignments page:
- Must include trips created by clinic that are in city_id == activeCityId and scheduled for that day (or next 24h)
- Show UNASSIGNED list with full rows (patient, times, pickup, dropoff) not just counts.
- Provide quick “Assign” from the list.

========================================================
PHASE 4 — DRIVER TRIP WORKFLOW (PROGRESS BUTTONS)
========================================================
Driver must have a clean progression, not only “Finish”.

Driver Trip Detail must show ONE primary button at a time:
- If ASSIGNED -> “Start Trip (En Route to Pickup)” sets EN_ROUTE_TO_PICKUP
- If EN_ROUTE_TO_PICKUP -> “Arrived at Pickup” sets ARRIVED_PICKUP
- If ARRIVED_PICKUP -> “Picked Up Patient” sets PICKED_UP
- If PICKED_UP -> “Arrived at Dropoff” sets ARRIVED_DROPOFF
- If ARRIVED_DROPOFF -> “Complete Trip” sets COMPLETED

Also include secondary actions:
- “Report No-Show” -> sets NO_SHOW (requires reason)
- “Cancel Trip” -> sets CANCELLED (requires reason) only if allowed by role rules

Backend:
- POST /api/driver/trips/:tripId/transition { toStatus }
Server-side transition rules (strict):
ASSIGNED -> EN_ROUTE_TO_PICKUP
EN_ROUTE_TO_PICKUP -> ARRIVED_PICKUP
ARRIVED_PICKUP -> PICKED_UP
PICKED_UP -> ARRIVED_DROPOFF
ARRIVED_DROPOFF -> COMPLETED
Any other transition = 400.

Also:
- Write timestamp fields on each transition (arrived_pickup_at, picked_up_at, etc.)
- These timestamps are used for compliance and ops health.
