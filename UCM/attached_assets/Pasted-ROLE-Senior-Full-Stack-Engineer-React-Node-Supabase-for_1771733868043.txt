PHASE 1 — DATABASE / SCHEMA (SAFE MIGRATIONS)
---------------------------------------------------------

1) Create table: trip_requests
Columns:
- id (uuid pk)
- tenant_id (uuid) OR company_id depending on existing tenancy model
- company_id (uuid, transport company that will service it)
- clinic_id (uuid)
- created_by_user_id (uuid)
- patient_id (uuid nullable if new)
- pickup_address (text)
- dropoff_address (text)
- pickup_lat (numeric) pickup_lng (numeric) nullable
- dropoff_lat (numeric) dropoff_lng (numeric) nullable
- pickup_time (timestamptz)
- service_level (text: ambulatory/wheelchair/stretcher)
- is_round_trip (bool)
- recurrence_rule (text nullable; for dialysis schedules)
- notes (text)
- status (text enum: 'PENDING_APPROVAL','NEEDS_INFO','APPROVED','REJECTED','CANCELLED')
- dispatch_notes (text nullable)
- approved_trip_id (uuid nullable; links to trips table when approved)
- created_at, updated_at

Index: (company_id, clinic_id, status, pickup_time)

2) Add/ensure patient associations:
If patients table already exists, enforce these columns (add if missing):
- company_id (uuid) NOT NULL (transport company)
- clinic_id (uuid) NOT NULL
- city_id (uuid) NOT NULL (or derivable from company)
Add index: (company_id, clinic_id, city_id)

3) Create chat tables:
- chat_threads
- id uuid pk
- scope_type text ('TRIP_REQUEST','TRIP','CLINIC_GENERAL')
- scope_id uuid
- company_id uuid
- clinic_id uuid
- created_at
Unique constraint: (scope_type, scope_id)

- chat_messages
- id uuid pk
- thread_id uuid fk
- sender_user_id uuid
- sender_role text
- message text
- created_at
Indexes: (thread_id, created_at)

4) RLS policies:
- Clinics can read/write only where clinic_id matches their clinic.
- Dispatch/company admins can read/write threads/messages for their company_id.
- NO cross-company access.
- Service role bypass only as needed.

PHASE 2 — BACKEND API / SERVICES
---------------------------------------------------------

Implement endpoints (or RPC) with RBAC checks:

Clinic side:
- POST /api/clinic/trip-requests (create request)
- GET /api/clinic/trip-requests (list)
- GET /api/clinic/trip-requests/:id (details)
- POST /api/clinic/patients (create patient with company_id + clinic_id + city_id)
- GET /api/clinic/patients (list/search filtered)
- POST /api/chat/threads (get-or-create thread by scope)
- GET /api/chat/threads/:id/messages
- POST /api/chat/threads/:id/messages

Dispatch side:
- GET /api/dispatch/trip-requests (list by status/company/city)
- POST /api/dispatch/trip-requests/:id/approve
Behavior:
- Create real trip in trips table using request data
- Set trip_requests.status='APPROVED'
- Save approved_trip_id
- POST /api/dispatch/trip-requests/:id/reject (store reason)
- POST /api/dispatch/trip-requests/:id/needs-info (set status and message)

IMPORTANT:
Approval must be idempotent: cannot create multiple trips for same request.
