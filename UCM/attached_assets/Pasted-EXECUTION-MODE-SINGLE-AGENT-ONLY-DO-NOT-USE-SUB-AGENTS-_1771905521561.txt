STEP 1 — Fix the immediate “Failed to fetch” (network/root cause)
Most common causes in this setup:
A) Frontend calls a hardcoded API base (e.g., appBaseUrl) that doesn’t match the current subdomain → CORS/blocked.
B) Reset endpoint path differs across portals → 404.
C) Reset uses fetch to a URL missing https or wrong host → fails.
D) Missing CORS headers if cross-origin is required.

Implement this rule:
- Frontend MUST call API using a single helper that is correct for the current deployment.
- If API is same-origin: use relative URLs only: fetch("/api/...") never absolute.
- If API is centralized on app.*: set ONE canonical API_BASE_URL and configure CORS on server to allow driver.* and clinic.*.

STEP 2 — Portal-aware reset link generation (the core requirement)
Goal: reset links must open on the correct subdomain based on user role/type.

Implement a deterministic “portal resolver”:
- DRIVER -> https://driver.unitedcaremobility.com
- CLINIC -> https://clinic.unitedcaremobility.com
- ADMIN/DISPATCH/SUPER_ADMIN -> https://app.unitedcaremobility.com

Where to implement:
- In the server function that creates/sends the reset email:
  - Determine user role/type from DB by email (or userId).
  - Choose portalBaseUrl = portalForRole(role).
  - Build link: `${portalBaseUrl}/login?token=${token}&type=recovery&reset=true`
    OR `${portalBaseUrl}/reset-password?token=${token}` (preferred for clarity).

If you are using Supabase reset emails:
- Do NOT rely on a single SiteURL for all portals.
- Instead pass redirectTo explicitly when generating the reset:
  supabase.auth.resetPasswordForEmail(email, { redirectTo: `${portalBaseUrl}/auth/callback?next=/reset-password` })
- Ensure each portal has the callback route implemented.

If you are using custom tokens:
- Create/confirm table password_reset_tokens:
  - id (uuid)
  - user_id
  - token_hash (sha256)
  - portal (enum: driver|clinic|app)
  - expires_at (e.g., now + 30 min)
  - used_at nullable
  - created_at
- Email contains raw token; DB stores only hash.
- Verify: hash(token) == token_hash AND used_at is null AND now < expires_at.

STEP 3 — Make reset work in BOTH modes
MODE 1: Reset via email link
- When portal loads with token query param:
  - Frontend calls POST /api/auth/reset/verify (token) → returns ok + user context + “allowReset”.
  - Then user enters new password → POST /api/auth/reset/confirm (token, newPassword).
  - Server validates token again, sets password, marks used_at.

MODE 2: Temporary password login (force change)
- If user logs in with temporary password:
  - Server issues session BUT with flag: must_change_password = true.
  - Frontend detects flag and forces Change Password screen.
  - On submit, call POST /api/auth/password/change with (currentTempPassword, newPassword).
  - Server validates current password, updates to new password, clears must_change_password.

Important:
- Do NOT mix temp-password reset with email token validation.
- Both should end in the same “set password” code path with shared validator.
